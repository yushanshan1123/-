# PROTOTYPE_TO_PRODUCTION_REPLACEMENT_PLAN.md

## 项目名称
新币合约交易冷静器 2.1

## 文档目标
这份文档用于回答一个非常实际的问题：

> **现在已经跑通的 MVP 原型闭环，下一步应该怎么逐步替换成正式调用链。**

重点不是再讨论“做什么功能”，而是明确：
- 哪些东西现在还是原型
- 它们未来应该被什么正式模块替代
- 替换顺序应该怎么排
- 哪些可以先保留，哪些必须优先替换

---

## 一、当前现状
当前已经跑通的原型闭环包括：

1. Binance 主骨架读取
2. `MarketSnapshot` 生成
3. live 风控
4. 提醒 → 风控
5. 提醒 → 速评
6. 风控 → 记录
7. 记录 → 复盘

这说明 MVP 原型闭环已经成立。  
但它目前仍然主要基于：
- Python 原型脚本
- 本地 JSON 文件
- CLI/模拟调用

所以它现在更准确的定位是：

> **可演示原型链，不是正式运行链。**

---

## 二、替换原则
正式替换时，建议遵守以下原则：

### 原则 1：先替换“调用方式”，再替换“能力本身”
也就是说：
- 先把脚本式调用，替换成正式函数 / 服务调用
- 不要一上来就重写业务逻辑

### 原则 2：先保留已经验证有效的逻辑
当前原型已经跑通，说明很多逻辑是有效的。  
正式化时应优先“封装”和“迁移”，而不是推倒重来。

### 原则 3：先替换主骨架，再替换增强层
先把：
- Binance
- 风控
- 记录
- 复盘

这些主骨架接稳。  
Funding / Heatmap / GMGN 仍然放后面。

### 原则 4：先替换链路，再优化体验
先确保：
- 数据真能流
- 调用真稳定
- 记录真能存

再考虑：
- UI
- 多入口优化
- 更漂亮的话术

---

## 四、P0：第一批必须替换的部分

## 1. `binance_connector_prototype.py`
### 当前作用
- 拉取 Binance 公共接口
- 生成最小 Connector 输出
- 映射 `MarketSnapshot`

### 问题
- 现在是脚本式原型
- 更适合演示，不适合正式链路长期依赖

### 正式替换目标
替换成：
- 正式 `BinanceConnector`
- 或正式 `market_data_service`

### 替换后应具备
- 标准函数入口
- 更清晰的错误处理
- 更稳定的日志与超时处理
- 可被业务层直接调用

---

## 2. `xinbi_live.py`
### 当前作用
- 快照读取
- 风控逻辑
- 风控结果组装

### 问题
- 当前已经很接近服务层，但还不是正式服务边界

### 正式替换目标
拆成：
- `market_snapshot_service`
- `risk_check_service`
- （后续）`review_service`

### 替换建议
不是推翻，而是把现有函数迁入更正式的模块目录结构。

---

## 3. `xinbi_cli.py`
### 当前作用
- 作为当前最标准的调用入口

### 问题
- 还是 CLI 入口
- 不能视作正式产品调用链

### 正式替换目标
替换成：
- Skill 运行时调用入口
- 或内部 API 层

### 替换意义
这一步完成后，系统就不再是“脚本演示”，而是“正式调用”。

---

## 4. `trade_review_records.json`
### 当前作用
- 保存交易计划记录
- 支撑复盘原型

### 问题
- 本地 JSON 文件只适合原型阶段
- 不适合多用户、并发、长期维护

### 正式替换目标
替换成：
- 正式数据库
- 或正式持久化存储层

### 这是 P0 的原因
因为只要记录层还是 JSON 文件，
整条闭环就很难算“正式可运行”。

---

## 1. `alert_to_risk_prototype.py`

### 正式替换目标

### 当前作用

### 正式替换目标

## 六、建议替换顺序
为了避免一边换一边散，建议按这个顺序替换：

### Phase A：底层正式化
1. 正式化 Binance Connector
2. 正式化 `MarketSnapshot` 服务
3. 正式化风控服务

### Phase B：存储正式化
4. 正式化记录写入层
5. 正式化记录查询层
6. 正式化复盘服务

### Phase C：入口正式化
7. 正式化提醒分流入口
8. 正式化 Skill 调用入口

### Phase D：增强层接入
9. Funding
10. Heatmap
11. GMGN（若可行）

---

## 七、当前最应该先替换什么
如果现在只问一句：

> **下一步第一个正式化动作是什么？**

答案是：

# 先把 `binance_connector_prototype.py` 和 `xinbi_live.py` 变成正式服务入口

原因：
- 它们是整个原型闭环的底座
- 不先替换底座，上层替换意义不大
- 一旦这两层正式化，后续提醒、风控、记录、复盘都能逐步迁移

---

## 八、当前阶段一句话结论
现在最正确的思路，不是再继续堆新的原型脚本，
而是：

> **把已经跑通的原型链，按“底层数据服务 → 风控服务 → 存储服务 → 分流入口”的顺序，逐步替换成正式调用链。**

这才是从“MVP 原型闭环成立”迈向“正式 MVP 可运行”的下一步。
